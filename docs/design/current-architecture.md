## 2. 实例配置文件详解

当前配置文件 `config.toml` 完整内容：

```toml
# ============================================
# 输入节点（网关入口）
# ============================================

[input.service]
port = 8000
models = ["qwen3.5-35b-a3b", "qwen3.5-122b-a10b"]

[input.service.alias]
"data/Qwen3.5-35B-A3B" = "qwen3.5-35b-a3b"
"Qwen3.5-35B-A3B" = "qwen3.5-35b-a3b"
"Qwen3.5-122B-A10B" = "qwen3.5-122b-a10b"

[input.test]
port = 8001
models = ["qwen3.5-35b-a3b", "qwen3.5-122b-a10b", "qwen3.5-35b-a3b-no-mtp"]

# ============================================
# 虚拟节点（路由策略）
# ============================================

[node."qwen3.5-35b-a3b"]
sequence = ["limit-sglang-qwen-35b", "aliyun"]

[node."qwen3.5-122b-a10b"]
sequence = ["limit-sglang-qwen-122b", "aliyun"]

[node.limit-sglang-qwen-35b]
concurrency = { max = 8, successor = "sglang-qwen3.5-35b-a3b" }

[node.limit-sglang-qwen-122b]
concurrency = { max = 8, successor = "sglang-qwen3.5-122b-a10b" }

# ============================================
# 后端节点（实际服务）
# ============================================

[backend]
"sglang-qwen3.5-35b-a3b" = "http://172.17.250.163:30001"
"sglang-qwen3.5-122b-a10b" = "http://172.17.250.163:30002"
"qwen3.5-35b-a3b-no-mtp" = "http://172.17.250.163:30003"

[backend.aliyun]
base-url = { anthropic = "https://dashscope.aliyuncs.com/apps/anthropic" }
api-key = "$ALIYUN_API_KEY"

# ============================================
# 健康监控与统计
# ============================================

[health]
window-size = 60
failure-threshold = 3
cooldown-duration = 300

[statistics]
enabled = true
db-path = "stats.db"
retention-days = 7

[admin]
port = 8080
```

### 配置结构说明

| 配置块         | 作用                             | 节点类型 |
|----------------|----------------------------------|----------|
| `[input]`      | 定义网关入口，监听端口和可用模型 | 输入节点 |
| `[node]`       | 定义模型的路由策略               | 虚拟节点 |
| `[backend]`    | 定义实际的后端服务地址           | 后端节点 |
| `[health]`     | 健康检查参数                     | 全局配置 |
| `[statistics]` | 统计存储配置                     | 全局配置 |
| `[admin]`      | 管理 API 配置                    | 全局配置 |

### 4.2 请求生命周期

```plaintext
阶段 1: 接收与解析
═══════════════════════════════════════════════════════════════════════════

  客户端请求
      │
      ▼
  ┌──────────────────────────────────────┐
  │ 1. HTTP 连接建立                     │
  │ 2. 解析请求头 (method, path, headers)│
  │ 3. 如果是 /v1/models，直接返回列表   │
  │ 4. 读取并解析请求体为 JSON           │
  │ 5. 提取 model 字段                   │
  │ 6. 识别协议 (OpenAI/Anthropic)       │
  └──────────────────────────────────────┘
      │
      ▼
  RoutePayload { protocol, model, parts, body }


阶段 2: 路由决策（节点图遍历）
═══════════════════════════════════════════════════════════════════════════

  RoutePayload
      │
      ▼
  ┌────────────────────────────────────────────────────────────────────┐
  │ InputNode                                                          │
  │ 1. 应用别名映射（如有）                                            │
  │ 2. 查找 model 对应的后继节点                                       │
  │ 3. 调用后继节点的 route()                                          │
  └───────────────────────┬────────────────────────────────────────────┘
                          │
                          ▼
              ┌─────────────────────┐
              │   虚拟节点层        │
              │                     │
              │ ┌─────────────┐     │
              │ │Concurrency  │─────┼──► 获取许可？否 ──► 返回 503
              │ │   Limit     │     │
              │ └──────┬──────┘     │
              │        │ 是         │
              │        ▼            │
              │ ┌─────────────┐     │
              │ │  Sequence   │─────┼──► 依次尝试后端
              │ │   Node      │     │    - 健康检查 ✓？
              │ └──────┬──────┘     │    - 路由成功 ──► 继续
              └────────┼────────────┘    - 失败 ──► 尝试下一个
                       │
                       ▼
              ┌─────────────────────┐
              │   后端节点层        │
              │                     │
              │ - 检查健康状态      │
              │ - 匹配协议与 URL    │
              │ - 确定是否需要转换  │
              └──────────┬──────────┘
                         │
                         ▼
              Route { guards, backend }


阶段 3: 请求转发
═══════════════════════════════════════════════════════════════════════════

  Route
    │
    ├──► 协议相同？
    │      ├──► 是 ──► 直接转发请求体
    │      └──► 否 ──► 转换请求体格式
    │                   (OpenAI ↔ Anthropic)
    │
    ├──► 添加/替换 API Key
    │
    ├──► 发送 HTTP 请求到后端
    │
    └──► 接收响应


阶段 4: 响应处理
═══════════════════════════════════════════════════════════════════════════

  后端响应
    │
    ├──► 检查响应状态
    │      ├──► 成功 ──► health.record_success()
    │      └──► 失败 ──► health.record_failure()
    │
    ├──► 流式响应？
    │      ├──► 是 ──► 包装 Body（保持 guards 存活）
    │      │           协议不同？
    │      │             ├──► 是 ──► 实时转换 SSE 帧
    │      │             └──► 否 ──► 直接透传
    │      └──► 否 ──► 一次性返回
    │
    └──► 返回给客户端


阶段 5: 资源释放
═══════════════════════════════════════════════════════════════════════════

  流式响应结束
      │
      ▼
  Body 被 drop
      │
      ▼
  NodeHoldingBody 释放
      │
      ▼
  RouteGuard 列表被 drop
      │
      ├──► ConcurrencyGuard drop ──► 并发计数 -1
      ├──► SimpleGuard drop ──► 无操作
      │
      ▼
  请求完全结束，资源全部释放
```

## 5. 核心设计模式

### 5.1 RAII 资源管理

```plaintext
                    请求开始
                       │
                       ▼
            ┌───────────────────────┐
            │ ConcurrencyGuard::new │
            │                       │
            │ 原子操作: count += 1  │
            │ 超过限制?             │
            │   ├──► 是: 回滚,返回 None
            │   └──► 否: 创建 Guard │
            └───────────┬───────────┘
                        │
                        ▼
            Guard 被添加到 Route.guards
                        │
            ┌───────────┴──────────┐
            │                      │
            ▼                      ▼
     请求处理中              流式传输中
     Guard 保持存活          Guard 保持存活
```

## 7. 重构关注点

### 7.1 Pingora 适配关键挑战

```plaintext
挑战 1: 并发控制保持到流结束
═══════════════════════════════════════════════════════════════════════════

当前实现:
  Hyper Body ──► NodeHoldingBody ──► 持有 guards ──► Drop 时释放

Pingora 适配:
  ProxyHttp::CTX ──► 需要验证生命周期 ──► 流结束才 Drop?

关键问题:
  - CTX 何时创建？何时销毁？
  - 能否在 CTX 中存储 Semaphore permit?
  - 流式响应期间 CTX 是否保持存活?


挑战 2: 策略链框架
═══════════════════════════════════════════════════════════════════════════

当前实现:
  SequenceNode ──► 顺序尝试 ──► 递归调用 route()

Pingora 适配:
  StrategyChain ──► 插件 trait ──► upstream_peer() 中执行

需要设计:
  - RoutingPlugin trait
  - StrategyChain 执行器
  - 错误处理与重试逻辑


挑战 3: 健康检查集成
═══════════════════════════════════════════════════════════════════════════

当前实现:
  BackendNode::route() ──► health.is_available()
  serve::handle_route_success() ──► health.record_success/failure()

Pingora 适配:
  ProxyHttp::logging() ──► 响应完成回调 ──► 记录健康状态

可行方案:
  - 在 CTX 中记录选中的后端
  - logging() 回调中根据成功/失败更新健康状态


挑战 4: 协议转换流式处理
═══════════════════════════════════════════════════════════════════════════

当前实现:
  NodeHoldingBody::poll_frame() ──► 实时转换 SSE 帧

Pingora 适配:
  Pingora Body trait ──► 需要确认转换方式

可选方案:
  - 使用 Pingora TransformBody API
  - 自定义 Body 实现
  - 流式转换器适配
```

## 8. 总结

当前 LLM Gateway 的核心设计亮点：

1. **虚节点图架构** - 通过 ConcurrencyNode、SequenceNode、BackendNode 的组合实现灵活的路由策略
2. **RAII 资源管理** - ConcurrencyGuard 确保并发计数精确控制到流结束
3. **健康检查熔断** - 滑动窗口 + 冷却期机制防止故障扩散
4. **协议自动转换** - 根据后端支持的协议自动进行 OpenAI ↔ Anthropic 转换

重构到 Pingora 时需要重点保留这些设计思想，同时适配新的框架 API。
